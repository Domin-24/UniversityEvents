# Team Communication Plan — ระบบจัดการกิจกรรมมหาวิทยาลัย

**อัปเดตล่าสุด:** 22 มีนาคม 2026

---

## 1. ข้อมูลทีม

| สมาชิก | ความรับผิดชอบหลัก |
|--------|-----------------|
| นางสาวกมลชนก เรืองแสน | ผู้กำหนดทิศทางและความต้องการของระบบ |
| พชรพล เชิดชู | ประสานงาน กำหนดเวลา ลบอุปสรรค |
| นายทิษณุ กลิ่นกำธรกุล | API, business logic, database queries |
| นายเทพเมธี ศรีพล | Schema design, migrations, Docker, CI/CD |
| นายรพีพัฒน์ ทับทอง | React UI, state management, API integration |

## 2. ช่องทางการสื่อสาร

| ช่องทาง | วัตถุประสงค์ | เวลาตอบ |
|---------|------------|---------|
| **LINE Group** | แจ้งข่าวด่วน, บล็อกเข้าทำงานไม่ได้, นัดหมายด่วน | ภายใน 2 ชั่วโมง |
| **Discord** | รายงาน bug, feature request, ติดตามงาน | ภายใน 1 วันทำการ |
| **Discord** | Code review, ข้อเสนอแนะเฉพาะโค้ด | ภายใน 2 วันทำการ |
| **Discord** | ถกเถียงออกแบบ, decision log | ภายใน 2 วันทำการ |
| **Discord** | Sprint planning, retrospective | ตามนัด |

---

## 3. ตารางการประชุม

| การประชุม | ความถี่ | ระยะเวลา | วันเวลา | ผู้รับผิดชอบ |
|-----------|--------|---------|---------|------------|
| **Daily Stand-up** | ทุกวัน | 15 นาที | 17:00 น. | Scrum Master |
| **Sprint Planning** | ทุก 2 สัปดาห์ | 1 ชั่วโมง | วันจันทร์ Sprint | Scrum Master |
| **Sprint Review** | ทุก 2 สัปดาห์ | 30 นาที | วันศุกร์ Sprint | ทุกคน |
| **Sprint Retrospective** | ทุก 2 สัปดาห์ | 30 นาที | วันศุกร์ Sprint | Scrum Master |
| **Code Review Session** | ตามต้องการ | 45 นาที | ตามนัด | Dev |

### รูปแบบ Daily Stand-up (3 คำถาม)
1. **เมื่อวานทำอะไร?** — งานที่ทำเสร็จ
2. **วันนี้จะทำอะไร?** — งานที่วางแผน
3. **มีบล็อกอะไรไหม?** — ปัญหาที่ต้องการความช่วยเหลือ

---

## 4. การแจ้งปัญหาและบล็อก

**ถ้าบล็อกทำงานไม่ได้:**
1. แจ้งใน LINE Group ทันที พร้อมอธิบายปัญหา
2. ถ้าไม่แก้ได้ภายใน 2 ชั่วโมง — แจ้ง Scrum Master โดยตรง

**ถ้าพบ Bug ใน production หรือ develop:**
1. เปิด GitHub Issue ทันที ใส่ label `bug` + `priority: high`
2. แจ้งใน LINE Group
3. Assign ให้ผู้รับผิดชอบทันที

---

## 5. การตัดสินใจ (Decision Log)

ทุกการตัดสินใจสำคัญทางเทคนิคต้องบันทึกใน **GitHub Discussion** หรือ commit message พร้อม context:

- อะไรที่ถูกตัดสินใจ
- ทำไมถึงตัดสินใจแบบนี้ (alternatives ที่พิจารณา)
- ผลกระทบที่คาดว่าจะเกิด
